# 1. Project Overview

**Delivery / Courier System** — это современная платформа для заказа и осуществления курьерских доставок. 
Проект решает задачи связывания клиентов, которым нужно отправить посылку, и курьеров, готовых выполнить доставку. 
Система позволяет отслеживать статусы заказа, рассчитывать стоимость на основе дистанции и веса, а также поддерживает систему рейтингов и отзывов для курьеров.

Главные возможности:
- Регистрация и авторизация (Клиенты, Курьеры, Администраторы).
- Создание заказа с учетом типа груза, срочности и адресов.
- Подбор доступных заказов для курьеров по геолокации.
- Изменение статусов доставки в реальном времени.
- Расчет стоимости и времени прибытия с помощью Google Maps API.
- Оценка курьера после успешной доставки.

# 3. Project Architecture

Проект построен по классической **N-Tier (многослойной) архитектуре**.

1. **Controller Layer (Слой контроллеров):**
   Принимает HTTP-запросы от Frontend, проверяет права доступа (через аннотации `@PreAuthorize`), вызывает нужные сервисы и возвращает DTO (Data Transfer Objects) клиенту.
   *Примеры: `OrderController`, `AuthController`.*

2. **Service Layer (Слой бизнес-логики):**
   Содержит всю "умную" логику приложения. Здесь происходят расчеты стоимости, проверки статусов, вызовы внешних API и взаимодействие с базой данных.
   *Примеры: `OrderService`, `AuthService`, `GoogleMapsService`.*

3. **Repository Layer (Слой доступа к данным):**
   Интерфейсы, наследующие `JpaRepository`. Отвечают за выполнение SQL-запросов к PostgreSQL с помощью Hibernate.
   *Примеры: `OrderRepository`, `UserRepository`.*

4. **Entity Layer (Слой моделей БД):**
   Java-классы (Entity), которые маппятся (проецируются) на таблицы в базе данных. Описывают структуру таблиц и связи между ними.
   *Примеры: `Order`, `User`, `Review`.*

5. **Security Layer (Слой безопасности):**
   Конфигурация `SecurityConfig`, фильтры `JwtAuthenticationFilter`. Отвечают за проверку токенов при каждом запросе и разграничение прав доступа (Role-Based Access Control).

6. **DTO & Mappers (Объекты передачи данных и мапперы):**
   Используются для отделения моделей БД от данных, которые отправляются клиенту. Это обеспечивает безопасность (не передаем пароли) и гибкость.

# 4. Database Structure

### 1. `User` (Пользователи)
- **Назначение:** Хранит всех пользователей системы (клиентов, курьеров, админов).
- **Поля:** `id`, `email`, `password`, `firstName`, `lastName`, `role`, `rating`, `reviewsCount` и др.
- **Связи:** 
  - `OneToMany` к `Order` (как клиент и как курьер).
  - `OneToMany` к `Review`.

### 2. `Order` (Заказы)
- **Назначение:** Главная сущность приложения. Содержит всю информацию о доставке.
- **Поля:** `id`, `pickupAddress`, `destinationAddress`, `price`, `distanceKm`, `status`, `cargoType`, `urgency` и координаты.
- **Связи:**
  - `ManyToOne` к `User` (client) — Заказ всегда принадлежит одному клиенту.
  - `ManyToOne` к `User` (courier) — Заказ может быть назначен на одного курьера.
  - `OneToMany` к `DeliveryStatusHistory` — История всех изменений статусов заказа.

### 3. `DeliveryStatusHistory` (История статусов доставки)
- **Назначение:** Трекинг каждого изменения статуса заказа для аналитики и отображения пользователю.
- **Поля:** `id`, `status`, `changedAt`.
- **Связи:**
  - `ManyToOne` к `Order` — Каждая запись истории принадлежит конкретному заказу.

### 4. `Review` (Отзывы)
- **Назначение:** Оценка клиентом работы курьера после доставки.
- **Поля:** `id`, `rating`, `comment`, `createdAt`.
- **Связи:**
  - `OneToOne` к `Order` — На один заказ можно оставить только один отзыв.
  - `ManyToOne` к `User` (client) и `User` (courier).

### 5. `CourierApplication` (Заявки на курьера)
- **Назначение:** Хранит заявки обычных пользователей, которые хотят стать курьерами (рассматриваются админом).

# 5. Authentication & Security

Безопасность реализована с использованием **Spring Security** и **JWT**.

- **Как работает Login / Register Flow:**
  1. Пользователь отправляет `POST /api/auth/login` с `email` и `password`.
  2. `AuthenticationManager` проверяет пароль (используется `BCryptPasswordEncoder`).
  3. Если данные верны, `JwtUtils` генерирует JWT токен и возвращает его клиенту.
  4. При последующих запросах клиент передает токен в заголовке `Authorization: Bearer <token>`.
  5. `JwtAuthenticationFilter` перехватывает запрос, проверяет валидность токена, извлекает ID пользователя и его Role, после чего кладет данные в `SecurityContextHolder`.

- **Roles & Permissions (Роли):**
  Существуют 3 основные роли: `CLIENT`, `COURIER`, `ADMIN`.
  Защита на уровне контроллеров происходит через аннотацию `@PreAuthorize("hasRole('ROLE_NAME')")`.

- **SecurityConfig:**
  Класс-конфигурация отключает CORS/CSRF (так как используется Stateless JWT подход), разрешает публичный доступ к `/api/auth/**` и Swagger, а остальные запросы требует аутентифицировать.

# 6. API Documentation

*Ниже приведены примеры самых важных эндпоинтов `OrderController`.*

### POST `/api/orders`
**Описание:** Создает новый заказ доставки.
**Roles:** `CLIENT`, `ADMIN`
**Request Body:** `pickupAddress`, `destinationAddress`, `pickupLat`, `pickupLon`, `destLat`, `destLon`, `cargoType`, `weight` и т.д.
**Response:** Данные созданного заказа (DTO) с просчитанной ценой и дистанцией.

### GET `/api/orders/available`
**Описание:** Возвращает список доступных заказов для курьеров (отсортированных по близости к курьеру).
**Roles:** `COURIER`
**Request Params:** `lat`, `lon`
**Response:** Список DTO заказов.

### POST `/api/orders/{id}/accept`
**Описание:** Курьер принимает доступный заказ. Статус меняется на `ASSIGNED`.
**Roles:** `COURIER`

### PATCH `/api/orders/{id}/pickup`
**Описание:** Курьер забрал посылку. Статус меняется на `PICKED_UP`.
**Roles:** `COURIER`

### POST `/api/orders/{id}/review`
**Описание:** Клиент оставляет отзыв на выполненный заказ.
**Roles:** `CLIENT`

# 7. Business Logic

Самый интересный флоу (Flow) работы с заказом происходит в `OrderService`:

1. **Создание заказа (`createOrder`):**
   - Клиент присылает координаты.
   - Вызывается внешний `GoogleMapsService`, который по координатам вычисляет точную дистанцию (км) и время в пути с учетом дорог.
   - Результат передается в `PricingService`, где рассчитывается итоговая `price` в зависимости от дистанции, веса груза (`weight`) и срочности (`urgency`).
   - Заказ сохраняется в БД со статусом `CREATED`, и создается первая запись в таблице истории (`DeliveryStatusHistory`).

2. **Поиск заказов курьером (`getAvailableOrdersForCourier`):**
   - Бэкенд достает из базы все заказы со статусом `CREATED`.
   - Для каждого заказа вычисляется расстояние от текущего местоположения курьера до точки отправления (с помощью **формулы Гаверсинуса** — `calculateHaversineDistance`).
   - Заказы сортируются от ближайшего к дальнему и отдаются на фронтенд.

3. **Принятие заказа (`acceptOrder`):**
   - **Важнейший момент:** используется **Pessimistic Locking** (`@Lock` в репозитории `findByIdForUpdate`). Это защищает от *Race Condition* (состояние гонки), чтобы два курьера не могли одновременно принять один и тот же заказ. Бэкенд на уровне БД блокирует строку заказа до завершения транзакции.
   - Также проверяется, что у курьера нет других активных заказов.

4. **Завершение и отзывы (`leaveReview`):**
   - После того как заказ перешел в статус `DELIVERED`, клиент может оставить отзыв.
   - Отзыв сохраняется в БД.
   - Рейтинг курьера в таблице `User` пересчитывается: берется старый рейтинг * количество отзывов + новая оценка, и делится на (количество отзывов + 1).

# 8. Exception Handling

В проекте реализован **Global Exception Handler** (`@ControllerAdvice`), который перехватывает все ошибки (Exception), возникающие в системе, и возвращает клиенту красивый JSON вместо стандартной ошибки Tomcat.

**Кастомные Exception:**
- `ResourceNotFoundException` — когда не найден пользователь, заказ и т.д. (возвращает статус 404).
- `IllegalStatusTransitionException` — если кто-то пытается перевести статус из `CREATED` сразу в `DELIVERED` (защита бизнес-логики).
- `CourierAlreadyHasActiveOrderException` — если курьер пытается взять два заказа одновременно.
- `ExternalApiException` — при сбоях в Google Maps.

# 9. Validation

Используется встроенный механизм `spring-boot-starter-validation` (Hibernate Validator).
В классах DTO (например, `OrderRequestDto`, `UserRegistrationRequest`) используются аннотации:
- `@NotBlank` — строка не должна быть пустой (например, для email или пароля).
- `@NotNull` — поле обязательно к заполнению.
- `@Min` / `@Max` — ограничения на вес посылки или рейтинг отзыва.

Если валидация не проходит, в контроллере автоматически (благодаря `@Valid`) выбрасывается `MethodArgumentNotValidException`, который перехватывается глобальным хэндлером и возвращает пользователю статус `400 Bad Request`.

# 10. External APIs

В проекте используется **Google Maps API** (через класс `GoogleMapsService`):
- **Для чего:** Вычисление точного расстояния между точкой А и точкой Б (Distance Matrix API).
- **Как работает:** `RestTemplate` отправляет HTTP-запрос на сервера Google, передавая координаты и API-ключ (сохраненный в переменных окружения). В ответ мы получаем точное расстояние в метрах и время в секундах, что используется для формирования цены доставки.

# 12. Possible Questions From Teacher

**В: Почему вы использовали JWT, а не сессии (Sessions)?**
*О:* У нас REST API и фронтенд на React. JWT позволяет сделать бэкенд *stateless* (без сохранения состояния). Нам не нужно хранить сессии в памяти сервера, что облегчает масштабирование приложения (можно легко поднять несколько инстансов бэкенда).

**В: Зачем нужны DTO (Data Transfer Object)? Почему не возвращать Entity напрямую?**
*О:* DTO нужны для безопасности и гибкости. Во-первых, если мы вернем `User` entity, мы случайно можем отдать пароль хэш (password hash) или служебные поля на фронтенд. Во-вторых, часто клиенту нужна сборная солянка данных из разных таблиц, и DTO позволяет сформировать нужный формат ответа независимо от структуры БД.

**В: Как вы решили проблему, если два курьера одновременно нажмут кнопку "Принять заказ"?**
*О:* В `OrderService.acceptOrder()` используется пессимистичная блокировка (Pessimistic Locking). Через репозиторий вызывается `findByIdForUpdate`, что на уровне PostgreSQL делает `SELECT ... FOR UPDATE`. Первый запрос заблокирует строку заказа, а второй будет ждать завершения транзакции первого, и затем выбросит ошибку, так как статус уже изменится на `ASSIGNED`.

**В: Для чего нужен Service Layer? Почему не писать логику прямо в Controller?**
*О:* Это принцип "Разделения ответственности" (Separation of Concerns). Контроллер должен только принимать запросы и возвращать ответы. Сервис должен содержать бизнес-логику. Это позволяет легко переиспользовать код сервисов (например, вызвать `OrderService` из планировщика задач или админки) и писать юнит-тесты без поднятия веб-контекста.

**В: Почему вы выбрали PostgreSQL?**
*О:* PostgreSQL — это мощная и надежная реляционная СУБД. У нас четкая структура данных со связями (Заказы, Пользователи, История), поэтому реляционная БД отлично подходит. Также она хорошо работает с гео-данными в будущем (через PostGIS), если проект будет развиваться.
