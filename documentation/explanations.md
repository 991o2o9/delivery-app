## 1. База данных и отношения (Relationships)

### Почему мы используем именно такие связи (Отношения) в базе данных?

В реляционных базах данных правильная настройка связей гарантирует целостность данных и убирает дублирование. Мы использовали **Spring Data JPA (Hibernate)** для проекции Java-классов в таблицы PostgreSQL.

**1. Связь `User` и `Order` (One-to-Many / Many-to-One)**
- **Почему `ManyToOne` внутри `Order`?** 
  В классе `Order` есть два поля: `client` и `courier`. Каждое из них имеет аннотацию `@ManyToOne`. Это значит, что **Много** заказов могут принадлежать **Одному** клиенту. И **Много** заказов могут быть выполнены **Одним** курьером. 
  Но конкретный заказ (1 штука) всегда имеет строго одного создателя (клиента) и строго одного исполнителя (курьера). Это классическая связь один-ко-многим со стороны пользователя.

**2. Связь `Order` и `DeliveryStatusHistory` (One-to-Many)**
- **Почему `OneToMany`?** 
  Жизненный цикл заказа состоит из статусов (`CREATED` -> `ASSIGNED` -> `PICKED_UP` -> `DELIVERED`). Нам нужно хранить **всю историю** изменений со временем (timestamp), чтобы показывать клиенту, где была его посылка в конкретное время. Поэтому **Один** заказ имеет **Много** записей в истории.

**3. Связь `Order` и `Review` (One-to-One)**
- **Почему `OneToOne`?** 
  По бизнес-логике на один доставленный заказ клиент может оставить только **один отзыв** курьеру. Если бы мы сделали `OneToMany`, клиент мог бы бесконечно спамить отзывы на один и тот же заказ, накручивая или сбивая рейтинг курьера. Связь "один к одному" защищает систему на уровне базы данных.

### Где находится подключение к БД и как оно работает?
Подключение к базе данных PostgreSQL настраивается в конфигурационном файле `backend/src/main/resources/application.properties` (или `.env`). Там прописываются `spring.datasource.url`, `username` и `password`.
При запуске приложения **Hibernate** читает классы с аннотацией `@Entity`, смотрит на их поля и связи, и автоматически (благодаря настройке `spring.jpa.hibernate.ddl-auto`) генерирует или обновляет SQL-таблицы.

## 2. Аутентификация: Почему JWT?

В проекте для аутентификации используется **JWT (JSON Web Token)**.

### Почему не классические сессии (Sessions)?
Обычно сайты сохраняют сессию пользователя в памяти сервера. Но у нас REST API и независимый Frontend на React (и, возможно, в будущем мобильное приложение).
**Плюсы JWT:**
1. **Stateless (Без состояния):** Серверу не нужно хранить в оперативной памяти информацию о том, кто сейчас "залогинен". Вся нужная информация (кто это и какая у него роль) зашифрована внутри самого токена.
2. **Масштабируемость:** Если проект вырастет, и мы запустим 5 серверов бэкенда, нам не придется синхронизировать сессии между ними. Любой сервер сможет расшифровать JWT токен.

### Как работает флоу аутентификации?
1. Пользователь вводит логин и пароль.
2. Сервер ищет пользователя в БД, берет зашифрованный пароль и сравнивает его с введенным (с помощью `BCrypt`).
3. Если всё верно, сервер берет **ID пользователя** и его **Роль**, подписывает их секретным ключом (который есть только у сервера) и генерирует строку-токен.
4. Сервер отдает токен фронтенду, и фронтенд сохраняет его (обычно в LocalStorage).

## 3. Безопасность и защита Эндпоинтов (Роли)

В системе есть 3 роли: `CLIENT`, `COURIER`, `ADMIN`.

### Как система понимает, что у человека есть доступ?

**Шаг 1: Перехват запроса фильтром**
При каждом запросе от фронтенда (например, курьер хочет взять заказ), фронтенд прикрепляет токен в заголовок:
`Authorization: Bearer <длинный_токен>`

В бэкенде есть специальный класс — `JwtAuthenticationFilter`. Он "стоит на воротах". Он ловит каждый HTTP-запрос ДО того, как тот попадет в контроллер.

**Шаг 2: Расшифровка и проверка**
Фильтр берет токен и с помощью секретного ключа проверяет его подпись. Если хакер попытался изменить роль в токене с `CLIENT` на `ADMIN`, подпись не совпадет, и сервер отклонит запрос (выдаст 401 Unauthorized).
Если токен валидный, фильтр достает из него **ID** и **Роль** (например, `ROLE_COURIER`).

**Шаг 3: Сохранение в контекст**
Фильтр кладет эти данные в специальный "сейф" текущего потока — `SecurityContextHolder`. Теперь сервер "помнит", кто делает этот запрос.

**Шаг 4: Защита в Контроллерах (@PreAuthorize)**
В контроллерах над методами стоят аннотации. Например:
```java
@PreAuthorize("hasRole('COURIER')")
@GetMapping("/available")
public ResponseEntity<List<AvailableOrderResponseDto>> getAvailableOrders(...)
```
Когда запрос доходит до этого метода, Spring Security "смотрит" в `SecurityContextHolder`. 
- Если там лежит роль `COURIER` — метод выполняется.
- Если там лежит `CLIENT` — Spring автоматически блокирует выполнение и возвращает ошибку **403 Forbidden** (Доступ запрещен).

Таким образом, безопасность работает автоматически, и нам не нужно в каждом методе писать `if (user.getRole() == COURIER)`.
